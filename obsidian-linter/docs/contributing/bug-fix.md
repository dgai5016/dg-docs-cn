# Bug 修复

在尝试修复 bug 之前，请确保已按照[配置指南](getting-setup.md)所述将 Linter 配置为本地可用。

将 Linter 作为本地仓库配置好之后，可以通过以下几种方式来修复 bug。

## 定位 Bug 的原因

当一个 bug 被上报或被你遇到，而你又不知道 bug 出在哪里时，可以尝试几种方法来缩小问题成因的范围。

### 加载会触发该 Bug 的配置

收到 bug 报告后，我通常会做的第一件事，就是确保自己拿到了上报者的 `data.json`。如果没有它，在某些情况下会很难复现问题。

### 逐个禁用规则

加载好配置后，下一步是开始禁用规则。我通常会一次只禁用一条规则，然后重新对相关文件 lint，看看能否复现问题。有时，如果我对某些规则与问题无关相当有把握，也会一次性关掉一批无关规则。

禁用规则的最终目标是把问题收敛到只剩一条规则。如果必须同时启用两条或更多规则才能复现问题，那这个 bug 的修复就会困难得多。

!!! note "多规则共同导致的 bug"
    当一个 bug 是由多条规则对文件依次做出修改而导致的，目前唯一能复现该行为的方式就是在 Obsidian 中进行。因此你需要先向 [rules-runner.ts](https://github.com/platers/obsidian-linter/blob/master/src/rules-runner.ts) 中添加 `console.log` 语句，然后运行 `npm run build`，借助每条启用的规则在处理文本时打印出的实际文本来辅助定位是哪条规则引发了 bug。

## 创建单元测试

如果 bug 由单条规则导致，请向该规则的[通用测试套件](testing.md#general-rule-test-suites)中[添加一个单元测试](testing.md#adding-tests)。这样既能更容易判断 bug 何时被修复，也有助于避免今后再次出现同类问题。

## 更新代码

接下来是有意思的部分：更新代码以修复 bug。有时这很直接，有时却很复杂。修复难度的关键在于该规则的现有逻辑有多复杂。

## 创建 pull request

代码修改完毕、bug 已修复后，即可[提交一个 pull request](open-a-pr.md)。
